§1  Executive Summary & Strategic Intent



The Knowledge Center Decision Engine (KCDE) is SHRM&apos;s internal AI reasoning infrastructure — a Graph-RAG system that converts two years of high-value Knowledge Center case resolution history into machine-executable intelligence. Its core function is to transition SHRM from a &apos;Search and Retrieval&apos; model (finding documents) to a &apos;Reasoning and Recommendation&apos; model (synthesizing answers grounded in SHRM&apos;s accumulated advisory expertise).



The KCDE is a reasoning engine, not a product. It is infrastructure — the layer that makes SHRM&apos;s Knowledge Center expertise computationally accessible, embeddable, and scalable. It surfaces through products and platforms: as an internal advisor augmentation tool for KC staff, as an API consumed by enterprise HRIS integrations, and eventually as a future integration point for the HR-X Solution Hub&apos;s Tier 2 compliance and policy guidance layer.



STRATEGIC UPDATE — Architecture Correction (1 of 2): ILS Positioning



The KCDE is no longer described as the &apos;primary Reasoning Layer&apos; or &apos;power grid&apos; of ILS, and is no longer positioned as a load-bearing component of the HR-X Solution Hub. The Hub&apos;s Tier 2 prescriptive layer operates independently; KCDE is acknowledged as a significant future integration that will deepen the compliance and policy guidance layer once it achieves production readiness. The Proprietary Intelligence Boundary principle is explicitly applied to all KCDE deployment modes: the vectorized BASK corpus and KC resolution corpus must reside in SHRM-owned infrastructure and are accessed via API only — not embedded in host platforms such as Microsoft Copilot or Workday.



STRATEGIC UPDATE — Architecture Correction (2 of 2): Phased Stack



The KCDE uses a phased technical architecture, matching the VoW Synthetic phasing principle. Phase 0 and Phase 1 operate on the Google stack: Qualtrics → BigQuery for corpus storage; Vertex AI Vector Search for BASK and resolution vectors; Gemini via Vertex AI for inference. This is consistent with the original v1.0 (March 25) architecture and keeps KCDE and VoW Synthetic on a single unified Phase 0 stack — reducing operational complexity, eliminating cross-cloud data movement, and keeping combined Phase 0 spend efficient. Phase 2+ migration to Microsoft-native infrastructure is triggered only by specific enterprise integration requirements (M365 Copilot deep integration, enterprise Azure API Management, SOC2 Type II at commercial scale) — not by default.

2.2  Contextual Reasoning Architecture



The KCDE uses a Context-Aware Graph-RAG architecture. Unlike standard retrieval-augmented generation, it does not simply retrieve a relevant document — it synthesizes a response grounded in the specific operational context of the user&apos;s environment, inherited at the moment of the query.

2.4  Operational Guardrails





Neutral Voice Control: All output uses strict third-person framing (&apos;An employer in California typically…&apos;) — no first-person legal advice. Prevents unauthorized practice of law.





Regulatory Conflict Flag: Engine surfaces a structured alert when historical resolution reasoning conflicts with a live regulatory update. Does not auto-resolve — human escalation path is triggered.





Human Escalation Path: High-stakes queries (multi-party employment disputes, complex leave intersections, termination scenarios in regulated environments) trigger mandatory handoff to a live KC Advisor. The KCDE never functions as a terminal endpoint for high-risk queries.





Deterministic Anchoring: BASK retrieval occurs before inference — not as a post-hoc filter. Prevents drift from SHRM standards in the reasoning path itself.

3.2  Phase 1 (Months 7–12) — Google Stack with Selective Additions



Phase 1 extends the Google stack to support Mode B (Agent-Ready Skill) and Mode C (Skilled Assistant) deployments. Selective additions introduced as specific use cases require them:





Agent discovery endpoint: SKILL.md manifest published at SHRM-owned URL for Mode B discovery by Microsoft Copilot, OpenClaw, and compatible orchestrators





Tool-use integration endpoints: ChatGPT Actions, Claude Tool Use, and Gemini Extensions integration for Mode C





Usage metering and rate limiting: Cloud Armor and API Gateway quota management for commercial access control





SOC2 readiness documentation: foundation for Phase 2 enterprise compliance if commercial demand materializes











3.3  Phase 2+ (Year 2+) — Microsoft-Native Migration (Conditional)



Phase 2 migration to Microsoft-native infrastructure is triggered only by specific enterprise integration requirements — not by default. The triggers for migration:





Mode D (API Service) demand from enterprise HRIS customers requiring native Azure API Management integration and M365 Copilot deep integration





SOC2 Type II compliance requirements at enterprise commercial scale that justify Microsoft Purview and Entra ID governance overhead





Enterprise customer data residency requirements that favor Azure&apos;s governance and tenant isolation capabilities





Consolidated infrastructure alignment with the HR-X Solution Hub&apos;s Phase 2 Microsoft-native migration (if Hub triggers first)



If none of these triggers materialize, KCDE remains on the Google stack indefinitely. Infrastructure decisions are driven by enterprise integration demand, not by architectural preference.

R6  Phase 3: Usability & UX Testing





Phase Status: FUTURE-GATED — requires working Phase 0 prototype of Mode A internal tool



Phase 3 usability testing is gated behind completion of the Phase 0 prototype — specifically the Mode A internal advisor augmentation tool, which is the first deployment mode that can be meaningfully tested with real users against real KC workflows. External mode testing (B, C, D) follows internal validation.



R6.1  Research Questions





Can a KC advisor, without training, use the Mode A interface to review an AI-drafted response, assess its accuracy, edit it where needed, and approve it for member delivery — within a time window that represents a net efficiency gain over the current manual process?





Where in the Mode A workflow do advisors stop trusting the AI draft and rewrite from scratch — and what specific output characteristics trigger that loss of trust?





Does the KCDE output for a sample of real-world question types meet the quality standards defined by advisors in Phase 2B?





For Mode C (branded assistant in consumer AI interfaces), can an HR professional ask a real HR question, receive a KCDE-grounded response, and correctly interpret when to act on it vs. when to escalate?







R6.2  Method

Parameter

Specification

Round 1: Mode A internal usability

n=6–8 KC advisors. Think-aloud moderated sessions. 60 minutes. Task scenarios: review and approve a correctly framed AI draft; identify and correct an error in an AI draft; handle a case where KCDE flags a Regulatory Conflict. Metrics: task completion rate; time-on-task vs. current baseline; error detection rate; SUS score.

Round 2: Mode C external usability

n=8–10 HR professionals from VoW Panel. Think-aloud. 45 minutes. Task scenarios: ask a jurisdiction-specific leave question; ask a termination procedure question; encounter a Human Escalation Path trigger and evaluate the handoff. Metrics: comprehension of response; action confidence rating; escalation trigger comprehension; SUS score.

Validation thresholds

Mode A: SUS ≥68; net time-per-case efficiency gain vs. current baseline; error detection rate ≥80% on seeded error cases. Mode C: correct action taken on AI response in ≥75% of scenarios; escalation trigger correctly interpreted in ≥85% of cases.

Timeline

Initiated at Phase 0 prototype completion — estimated M4–M6 of Phase 0 build.
